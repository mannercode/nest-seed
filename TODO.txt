1. libs/testing을 libs/common나 libs/microservices에 의존하지 않도록 개선.
   개선에 어려운 부분은 사전에 보고할 것
2. libs가 testing -> microservices -> common 이렇게 의존성을 가지는지 면밀하게 점검.
3. 그 외에 순홤참조 관련 문제 없는지 점검



- temporal 적용 후 검토는 하지 못한 상태
    - temporal 코드 모두 리뷰하고 libs/common에 통합 가능한지 검토해야 한다
    - temporal 도입하면서 sagaId는 어떻게 할지 재검토 필요
- sagaId를 모든 엔티티에 기본으로?
- 아래 기술들 추가할거 없나 검토
    1. 컨테이너/오케스트레이션
       • Docker: 서비스 패키징 표준
       • Kubernetes: 배포/스케일/서비스 디스커버리의 사실상 표준 플랫폼 ￼

    2. API Gateway / Edge

    외부 트래픽을 단일 진입점으로 받아서 라우팅/인증/레이트리밋/관측을 처리
    • 예: Kong, Apigee, NGINX, Envoy 기반 게이트웨이 등 ￼ 3. 서비스 메시(Service Mesh)

    서비스 간 트래픽을 “인프라 레벨”에서 제어(mTLS, 라우팅, 리트라이/타임아웃, 관측)
    • 예: Istio(Envoy 사이드카 기반), Linkerd, Consul Connect
    • Envoy가 메시/프록시 생태계의 핵심 구성요소로 자주 등장 ￼ 4. 메시징/이벤트 스트리밍(비동기 통신)

    MSA에서 동기 HTTP만 쓰면 결합도가 올라가서, 이벤트 기반이 흔합니다.
    • 예: Kafka, RabbitMQ, NATS, Pulsar ￼ 5. 관측성(Observability)

    MSA는 분산이라 “로그/메트릭/트레이싱” 3종 세트가 사실상 필수입니다.
    • OpenTelemetry(계측 표준), Jaeger(분산 트레이싱), Prometheus/Grafana(메트릭) ￼ 6. 설정/비밀(Secrets) 관리

    서비스가 많아질수록 .env로 버티기 어렵습니다.
    • 예: HashiCorp Vault(시크릿 라이프사이클/접근제어/감사 추적) ￼
    • (K8s라면) Kubernetes Secrets + 외부 KMS/Vault 연동 패턴도 흔함 7. CI/CD & GitOps

    배포 자동화와 환경 일관성이 중요해져서 GitOps가 자주 붙습니다.
    • 예: Argo CD(Kubernetes용 GitOps CD, desired state를 Git과 비교/동기화) ￼
    • (대안) Flux, Jenkins/GitHub Actions/GitLab CI 등 8. (옵션) 서비스 레지스트리/디스커버리

    K8s면 기본 DNS/Service로 해결되는 경우가 많지만, 비-K8s거나 멀티클러스터에서 별도 솔루션을 쓰기도 합니다.
    • 예: Consul, Eureka 등

    ⸻

    “NestJS + Temporal” 기준으로 현실적인 기본 세트(많이 쓰는 조합)
    • Kubernetes + API Gateway + (필요시) Service Mesh
    • Kafka(또는 RabbitMQ)
    • OpenTelemetry + Prometheus/Grafana + Jaeger
    • Vault(또는 K8s Secrets + KMS)
    • Argo CD(GitOps)
    Temporal은 여기에 “사가/장기 워크플로우”를 담당하는 엔진으로 들어가는 그림입니다.
