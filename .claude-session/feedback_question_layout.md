코드 검토 시, 파일 안의 결함뿐 아니라 *layout*과 *의존 방향*을 능동적으로 의심한다. 자연스러운 질문 예:

- 이 파일/심볼이 *이 패키지*에 있는 게 맞나? 더 적합한 자리는?
- 이 helper의 *실제 소비자*는 누구인가? 그들과 같은 패키지에 있나?
- 두 패키지가 같은 jest config / global setup / 의존성을 공유하는데, 그게 *둘 다 그게 필요해서*인가 아니면 *한 쪽이 끌려와서*인가?
- 모듈의 의존 방향이 자연스러운가? (env 생성자 → 소비자 helper가 같은 패키지인가)
- 어떤 패키지가 자기에게 필요 없는 비용(컨테이너 부팅 등)을 치르고 있지 않나?
- 두 패키지를 동시 실행하면 race가 나지 않나?

**Why:** 사용자 피드백 — `libs/testing`이 mongo/redis/s3/nats/temporal 컨테이너를 부팅하는데 자기 테스트엔 그 인프라가 일절 필요 없는 잘못된 jest config 공유를 사용자가 IDE에서 jest.global.js 한 번 열어보고 즉시 짚었다. 정적 도구로 안 잡히는 조직적 결함은 능동적 의문 없이는 안 보인다.

**How to apply:** 검토 라운드 중 한 번은 코드 안이 아니라 "이 파일이 왜 여기 있는가"를 본다. Explore 에이전트 프롬프트에도 "이 layout이 합리적인가"를 명시 항목으로 포함한다.
